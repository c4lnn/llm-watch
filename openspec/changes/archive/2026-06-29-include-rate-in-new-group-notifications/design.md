## Context

当前轮询流程会把适配器返回的分组归一化为 `{ key, name, rate, raw }`。新增分组检测发生在 `detectChanges()`：当当前分组不在上一轮快照中时，会生成 `available` 变更。但这个变更对象目前只包含 `group_id`、`group_name` 和 `type`，随后 `recordChanges()` 只能拼出 `🟢 新增: <分组名>`，无法在通知正文中展示倍率。

倍率已经在当前轮询结果里存在，并且快照写入时也会持久化到 `group_snapshots.rate`。本变更只需要把新增分组当次的 `group.rate` 传递到通知正文，不需要改变 adapter、通知渠道或数据库 schema。

## Goals / Non-Goals

**Goals:**
- 新增分组通知正文包含新增分组的当前倍率。
- 保持下线通知和字段变更通知现有语义与格式。
- 保持 `group_changes` 表结构和变更类型不变。
- 增加针对新增分组通知正文的测试覆盖。

**Non-Goals:**
- 不改变倍率变更的检测方式。
- 不为 `group_changes` 新增倍率列。
- 不调整 Bark 渠道 URL 格式、通知声音或通知分组。
- 不改变 dashboard 展示逻辑。

## Decisions

1. 在 `available` 变更对象中携带当前分组倍率。

   `detectChanges()` 已经遍历当前分组 map，能够直接读取 `group.rate`。把倍率附加到 `available` 变更对象，是最小且最贴近数据来源的修改。

   备选方案：在 `recordChanges()` 中重新查询最新快照获取倍率。该方案会增加数据库依赖，而且新增变更记录发生在快照写入前，时序更脆弱。

2. 只把倍率用于通知正文，不改变 `group_changes` 的持久化结构。

   当前 `group_changes` 记录的是变更类型、字段名和字段旧新值。新增分组的倍率是通知展示上下文，不是一个新的字段变更。因此不需要引入新列或新变更类型。

   备选方案：把倍率写入 `new_value`。这会让 `available` 记录混入字段变更语义，后续读取历史时更容易产生误解。

3. 使用稳定、简短的中文展示格式。

   新增分组行建议变为 `🟢 新增: <分组名> · 倍率: <rate>`。这和字段变更行中使用 `分组名 · 字段: 值` 的结构一致，便于手机通知快速扫读。

## Risks / Trade-offs

- 倍率为空或非数字时通知显示异常 -> 实现时应格式化倍率，无法得到有效倍率时可显示原始值或跳过倍率后缀。
- 测试只覆盖内部正文拼接而不触发真实 Bark -> 用单元测试覆盖正文格式即可，真实渠道发送逻辑不变。
- 首次轮询会把所有分组视为新增 -> 首次轮询通知会多显示倍率，这是符合本需求的行为。

## Migration Plan

无需数据迁移。部署后新的轮询通知立即使用新格式；既有历史变更记录保持不变。
